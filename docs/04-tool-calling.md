# 第 4 章：工具调用与 Function Calling

> Agent 与外部世界交互的核心技术。2025 年 MCP 和 A2A 协议成为新的高频考点。
> 内容来源：掘金、CSDN、腾讯技术社区、A2A 官方文档等。

---

## 一、核心知识点总结

### 1.1 Function Calling 原理

Function Calling 允许开发者向 LLM 描述一组函数签名，LLM 在需要时**决定调用哪个函数并生成 JSON 参数**，但**不实际执行**。开发者在本地执行函数后将结果返回给 LLM。

### 1.2 MCP（Model Context Protocol）

- Anthropic 提出的开放标准，被称为 "AI 的 USB 接口"。
- 标准化 LLM 与外部数据源/工具的双向通信。
- 解决各模型工具集成标准不统一的问题。

### 1.3 A2A（Agent-to-Agent Protocol）

- Google 2025 年推出，已移交 Linux Foundation 开源。
- 实现不同 AI Agent 之间的标准化通信与协作。
- 基于 JSON-RPC 2.0 over HTTP(S)。
- 引入 "Agent Card"（智能体名片）机制。

### 1.4 MCP vs A2A 的关系

| 维度 | MCP | A2A |
|------|-----|-----|
| 定位 | Agent ↔ 工具/数据 | Agent ↔ Agent |
| 类比 | 人使用工具 | 人与人协作 |
| 关系 | **互补，非竞争** | **互补，非竞争** |

## 二、高频面试题

### Q1：Function Calling 的原理和工作流程是什么？（⭐⭐⭐⭐⭐）

**参考答案**：

**工作流程**：
1. 开发者定义函数签名（名称、参数、功能描述）并传给 LLM。
2. 用户发出请求。
3. LLM 分析请求，判断是否需要调用函数。
4. 如需调用，LLM 返回函数名和 JSON 格式参数。
5. 开发者在应用层（本地）执行该函数。
6. 将执行结果再送回 LLM。
7. LLM 结合结果生成最终自然语言回答。

**关键点**：LLM 不实际执行函数，只完成**意图识别**和**参数提取**。

### Q2：什么是 MCP 协议？它解决了什么问题？（⭐⭐⭐⭐⭐）

**参考答案**：

MCP（Model Context Protocol）是 Anthropic 提出的开放协议，可以比喻为"AI 的 USB 接口"。

**解决的核心问题**：
- 过去每个 AI 模型与每个工具之间都需要定制化集成（M×N 问题）。
- MCP 统一了通信标准，使得任何 MCP 兼容的 LLM 都能连接任何 MCP 兼容的工具。
- 将 M×N 集成简化为 M+N。

**核心功能**：
1. **工具发现**：Agent 自动发现可用工具。
2. **参数协商**：标准化的参数描述和传递方式。
3. **安全管控**：内置权限和访问控制。

### Q3：什么是 A2A 协议？它和 MCP 的区别是什么？（⭐⭐⭐⭐⭐）

**参考答案**：

A2A（Agent-to-Agent Protocol）由 Google 推出，解决**不同 Agent 之间的通信协作**问题。

**核心特性**：
1. **Agent Card（智能体名片）**：每个 Agent 发布自己的能力描述，供其他 Agent 发现和选择。
2. 基于 JSON-RPC 2.0 over HTTP(S)，复用成熟的 Web 基础设施。
3. 支持认证授权机制，确保通信安全。

**与 MCP 的区别**：
- **MCP**：Agent 如何使用**工具**（类比：人使用锤子）。
- **A2A**：Agent 如何与**其他 Agent** 协作（类比：人与人合作）。
- **关系**：互补，不是竞争。在一个完整的多 Agent 系统中，A2A 用于 Agent 间通信，每个 Agent 内部通过 MCP 调用工具。

### Q4：Function Calling 的安全性如何保障？（⭐⭐⭐⭐）

**参考答案**：

1. **权限控制（Scoped Permissions）**：限定 Agent 可调用的函数范围。
2. **参数验证**：对 LLM 生成的参数进行格式和值域校验。
3. **人工确认（Human-in-the-Loop）**：对关键操作（如支付、删除）需用户二次确认。
4. **沙箱执行**：代码执行类工具在隔离环境中运行。
5. **审计日志**：记录所有函数调用的详细日志。
6. **回滚机制**：支持操作撤销和状态恢复。
7. **Prompt 注入防护**：防止恶意输入操纵函数调用。

### Q5：如何处理工具调用失败？（⭐⭐⭐⭐）

**参考答案**：

1. **超时控制**：为每次工具调用设置合理的超时时间。
2. **重试策略**：指数退避重试（Exponential Backoff）。
3. **降级方案（Fallback）**：主工具失败时切换到备用工具或直接用 LLM 知识回答。
4. **熔断机制（Circuit Breaker）**：连续失败达到阈值后暂停调用。
5. **幂等性设计**：确保重试不会产生副作用。
6. **错误反馈给 LLM**：将错误信息返回给 LLM，让其调整策略。

### Q6：Function Calling 是如何训练/微调的？（⭐⭐⭐⭐）

**参考答案**：

1. **训练数据构建**：构造（用户指令, 函数签名列表, 期望调用结果）三元组。
2. **格式**：通常将函数签名以 JSON Schema 的形式嵌入系统提示词。
3. **微调方式**：基于 SFT，让模型学会在需要时输出结构化的函数调用 JSON。
4. **开源方案**：Gorilla、Toolbench 等提供了 Function Calling 训练数据集。
5. **关键挑战**：高质量训练数据的构建是核心难点。

### Q7：MCP 协议有哪些潜在的安全风险？（⭐⭐⭐）

**参考答案**：

1. **工具注入攻击**：恶意工具通过 MCP 暴露有害功能。
2. **数据泄露**：Agent 通过工具访问敏感数据后外传。
3. **越权操作**：Agent 超出预设权限范围调用工具。
4. **中间人攻击**：通信过程被截获或篡改。

**防护措施**：TLS 加密、最小权限原则、工具签名验证、行为审计。

### Q8：请描述一个 MCP + A2A 协同使用的实际场景。（⭐⭐⭐⭐）

**参考答案**：

**场景：国际旅行规划**
1. 用户向"旅行规划 Agent"提出需求。
2. 旅行规划 Agent 通过 **A2A 协议**联系"机票 Agent"、"酒店 Agent"、"签证 Agent"。
3. 每个子 Agent 通过 **MCP 协议**调用各自的工具：
   - 机票 Agent → MCP → 航空公司 API
   - 酒店 Agent → MCP → 酒店预订 API
   - 签证 Agent → MCP → 签证信息数据库
4. 各 Agent 通过 A2A 将结果返回给规划 Agent，完成综合行程方案。
