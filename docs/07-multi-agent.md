# 第 7 章：多 Agent 系统

> 2025 年多 Agent 系统从学术概念走向生产核心。本章涵盖架构设计、框架对比、通信协作。
> 内容来源：CSDN、腾讯技术社区、掘金、百度开发者等。

---

## 一、核心知识点总结

### 1.1 为什么需要多 Agent

单 Agent 的局限性：
- 上下文窗口有限，难以同时管理多领域知识。
- 单一角色容易产生偏见。
- 复杂任务的各环节需要不同的专业能力。

### 1.2 多 Agent 架构模式

| 模式 | 核心思想 | 适用场景 |
|------|----------|----------|
| **中心化编排** | 一个管理 Agent 分配任务给子 Agent | 明确流程的自动化任务 |
| **去中心化协商** | Agent 通过能力广播和任务拍卖机制协调 | 开放性问题协作 |
| **层次化** | 多层管理结构 | 大型组织级任务 |
| **管道式** | Agent 按顺序依次处理 | 流水线式任务 |
| **辩论式** | 多 Agent 对同一问题提出不同观点 | 需要多角度分析的决策 |

### 1.3 主流框架对比

| 框架 | 设计哲学 | 核心特点 | 学习成本 |
|------|----------|----------|----------|
| **LangGraph** | 图结构 + 状态机 | 极致可控、状态持久化、HITL | 中 |
| **AutoGen** | 多Agent对话 | 灵活、动态协作、代码生成 | 中高 |
| **CrewAI** | 角色化团队协作 | 开箱即用、高度封装 | 低 |

业界简要总结：**"AutoGen 快、CrewAI 稳、LangGraph 强"**。

## 二、高频面试题

### Q1：为什么需要多 Agent 系统？相比单 Agent 有什么优势？（⭐⭐⭐⭐⭐）

**参考答案**：

1. **专业化分工**：每个 Agent 专注擅长领域（如一个写代码、一个做测试、一个做安全审查），提升整体质量。
2. **减少幻觉和偏见**：不同角色的 Agent 可以互相 Review、辩论、纠错。
3. **上下文隔离**：每个 Agent 有独立的上下文空间，避免单 Agent 的 Context Window 过载。
4. **并行提速**：独立的子任务可以并行执行。
5. **可扩展性**：新增能力只需添加新的专业 Agent，无需重构整个系统。
6. **容错性**：单个 Agent 失败不影响整体系统运行。

### Q2：LangGraph、AutoGen、CrewAI 三者的核心区别和选型建议？（⭐⭐⭐⭐⭐）

**参考答案**：

**LangGraph**：
- 基于有向图建模 Agent 工作流，每个节点是 Agent/工具调用，边是状态转换条件。
- **强项**：极致可控、状态持久化（支持中断恢复）、可观测性强。
- **适合**：生产级企业应用（银行审批、风控、合规流程）。

**AutoGen**：
- 基于多 Agent 对话机制，Agent 间类似群聊讨论。
- **强项**：灵活的协作模式、支持代码执行和验证。
- **适合**：研究探索、代码生成、数据分析。

**CrewAI**：
- 模拟人类团队，通过角色（Role）、任务（Task）、流程（Process）驱动。
- **强项**：封装度高、上手快、代码量少。
- **适合**：快速原型、内容营销、市场调研。

**选型建议**：
- 需要强流程控制和生产安全性 → **LangGraph**
- 需要 Agent 间深度讨论和迭代 → **AutoGen**
- 快速搭建 MVP 和团队协作调度 → **CrewAI**

### Q3：设计一个多 Agent 系统需要考虑哪些核心组件？（⭐⭐⭐⭐⭐）

**参考答案**：

1. **Agent 运行时（Agent Runtime）**：每个 Agent 的独立沙箱，包含 LLM、工具集和记忆。
2. **通信总线（Communication Bus）**：标准化的消息传递机制。
3. **编排引擎（Orchestration Engine）**：负责任务分解、Agent 分配和结果聚合。
4. **观察者（Observer）**：监控 Agent 状态，检测死锁和异常，触发熔断。
5. **共享记忆/黑板（Shared Memory/Blackboard）**：Agent 间共享信息的存储区。
6. **结果聚合器**：汇总各 Agent 输出，生成最终结果。

### Q4：多 Agent 系统中如何处理 Agent 间的冲突和一致性？（⭐⭐⭐⭐）

**参考答案**：

1. **投票机制**：多个 Agent 各自给出观点，多数投票决定。
2. **仲裁 Agent**：设立专门的判官 Agent 评估各方观点。
3. **层次化决策**：高层 Agent 对低层冲突做最终裁决。
4. **辩论+收敛**：让 Agent 展开辩论，直到观点收敛或达成共识。
5. **置信度加权**：根据每个 Agent 在对应领域的可靠度加权决策。

### Q5：多 Agent 系统的评估指标有哪些？（⭐⭐⭐⭐）

**参考答案**：

1. **任务成功率**：整体任务完成的正确率。
2. **协作效率**：Agent 间通信开销与任务完成速度比。
3. **输出一致性**：多次运行结果的稳定性。
4. **Token 消耗**：所有 Agent 合计的 Token 使用量。
5. **故障恢复能力**：单个 Agent 失败后系统的恢复速度。
6. **可扩展性**：增加 Agent 数量时的性能变化。

### Q6：设计一个自动代码评审（Code Review）的多 Agent 系统。（⭐⭐⭐⭐⭐）

**参考答案**：

**Agent 角色设计**：
1. **Coordinator Agent**：接收 Git Webhook 的 PR 通知，获取 Diff，分配任务。
2. **Security Agent**：检查安全漏洞（SQL 注入、XSS、硬编码密钥等）。
3. **Performance Agent**：检查性能问题（O(N²) 复杂度、内存泄漏等）。
4. **Style Agent**：检查代码风格、命名规范。
5. **Logic Agent**：检查业务逻辑正确性。
6. **Summary Agent**：汇总所有 Agent 意见，格式化为 Markdown 评审报告。

**工作流程**：
1. Coordinator 监听 Webhook、提取 Diff。
2. 多个审查 Agent **并行**分析各自维度。
3. Summary Agent 汇总并生成结构化报告。
4. 通过 GitHub API 将评论回写到 PR。

**关键设计**：
- Diff 过大时分文件/模块拆分。
- 设置置信度阈值，低置信度的问题标记为"建议"而非"错误"。
- 人工可覆盖 Agent 的审查意见。

### Q7：多智能体强化学习（MARL）的核心概念有哪些？（⭐⭐⭐）

**参考答案**：

1. **马尔可夫博弈（Markov Game）**：多 Agent 环境的数学框架。
2. **协作 vs 竞争**：Agent 间可以是合作关系、竞争关系或混合关系。
3. **CTDE（Centralized Training, Decentralized Execution）**：训练时用全局信息，执行时各自独立决策。
4. **通信学习**：Agent 学会互相发送有意义的消息。
5. **信用分配**：在团队奖励中确定每个 Agent 的贡献度。

### Q8：2025 年多 Agent 系统的发展趋势有哪些？（⭐⭐⭐⭐）

**参考答案**：

1. **协议标准化**：MCP（Agent↔工具）、A2A（Agent↔Agent）、AG-UI（Agent↔前端）等标准协议正在形成生态。
2. **通用 AI Agent 产品化**：能自主规划、持续执行的通用 Agent（如 Devin、OpenAI Operator）出现。
3. **Agent 即服务（AaaS）**：Agent 作为微服务部署，通过 API 组合。
4. **安全与合规**：Agent 间通信加密、身份认证、审计日志成为必备。
5. **人机协作深化**：关键节点设置人工审核，而非全自动。
