## 一、核心知识点总结

### 1.1 推理技术体系

| 技术 | 核心思想 | 论文/来源 |
|------|----------|-----------|
| Chain-of-Thought (CoT) | 引导模型逐步推理 | Wei et al., 2022 |
| Zero-shot CoT | 加一句"Let's think step by step" | Kojima et al., 2022 |
| Tree-of-Thought (ToT) | 多条推理路径并行探索 | Yao et al., 2023 |
| ReAct | 推理与行动交替 | Yao et al., 2022 |
| Reflexion | 执行后自我反思改进 | Shinn et al., 2023 |
| Self-Refine | 模型自我迭代优化输出 | Madaan et al., 2023 |

### 1.2 任务分解策略

- **子目标分解**：将复杂目标拆解为子目标序列。
- **递归分解**：逐层细化，直到子任务足够简单。
- **基于先验的分解**：利用领域知识预定义分解模板。

## 二、高频面试题

### Q1：简述 ReAct 框架的执行流程和优势。（⭐⭐⭐⭐⭐）

**参考答案**：

ReAct = Reasoning + Acting，交替进行推理和行动：

```
Thought: 我需要查找2024年诺贝尔物理学奖得主
Action: search("2024 Nobel Physics Prize winner")
Observation: Geoffrey Hinton and John Hopfield...
Thought: 找到了，现在可以回答用户
Action: finish("2024年诺贝尔物理学奖由 Geoffrey Hinton 和 John Hopfield 获得")
```

**优势**：
1. **可解释性**：每步都有思考过程记录。
2. **动态调整**：根据观察结果灵活修改下一步策略。
3. **减少幻觉**：通过工具调用获取真实信息而非臆造。
4. **适用范围广**：适配各种需要工具调用的场景。

### Q2：CoT 的常见做法和适用场景？（⭐⭐⭐⭐⭐）

**参考答案**：

**常见做法**：
1. **Zero-shot CoT**：末尾加 "Let's think step by step"。
2. **Few-shot CoT**：提供带推理过程的示例。
3. **Auto CoT**：自动生成推理链示例。

**适用场景**：
- 数学推理和多步计算。
- 复杂逻辑判断。
- 需要因果推理的问题。

**不适用场景**：
- 简单的事实查询。
- 信息检索类问题。
- 模型规模较小时效果不稳定。

### Q3：Tree-of-Thought 和 Chain-of-Thought 的核心区别？（⭐⭐⭐⭐）

**参考答案**：

| 维度 | CoT | ToT |
|------|-----|-----|
| 推理路径 | 单条线性链 | 多条并行分支 |
| 搜索策略 | 无回溯 | BFS/DFS，可回溯 |
| 评估机制 | 无中间评估 | 每步评估，剪枝无效路径 |
| 成本 | 低（1次 LLM 调用） | 高（多次 LLM 调用） |
| 适用场景 | 一般推理 | 创意写作、博弈、复杂规划 |

### Q4：什么是 Reflexion？它如何提升 Agent 性能？（⭐⭐⭐⭐）

**参考答案**：

Reflexion 让 Agent 在执行失败后进行**自我反思**，将反思结果存入记忆，指导后续尝试。

**流程**：
1. **执行**：Agent 尝试完成任务。
2. **评估**：判断是否成功。
3. **反思**：分析失败原因，生成反思摘要。
4. **重试**：将反思作为额外上下文，再次尝试。

**效果**：在编程任务（HumanEval）上，Reflexion 将 Pass@1 从 80% 提升到了 91%。

### Q5：Agent 的任务分解策略有哪些？各自适用什么场景？（⭐⭐⭐⭐）

**参考答案**：

1. **LLM 直接分解**：让 LLM 直接将复杂任务拆分为子任务列表。
   - 适用：一般性任务。
2. **递归分解**：逐层细化，子任务还可以继续拆分。
   - 适用：层次化的项目管理。
3. **基于工具的分解**：根据可用工具的能力来决定分解方式。
   - 适用：明确工具边界的场景。
4. **HuggingGPT 风格**：先分解为子任务，再为每个子任务匹配最合适的模型/工具。
   - 适用：多模态、多模型协作场景。

### Q6：如何设计高效的 Action Space（动作空间）？（⭐⭐⭐⭐）

**参考答案**：

设计高效 Action Space 的要点：
1. **粒度适中**：动作不能太粗（一步完成所有事）也不能太细（每步只做微操作）。
2. **原子性**：每个动作应该是一个完整的原子操作。
3. **正交性**：动作之间尽量不重叠。
4. **可组合性**：简单动作可以组合完成复杂任务。
5. **清晰的格式化**：每个动作的输入输出格式应该明确。
6. **Fallback 动作**：包含"我不知道"或"请求人类帮助"的兜底选项。

### Q7：如何让 Agent 具备自我纠错能力？（⭐⭐⭐⭐）

**参考答案**：

1. **输出验证**：每次执行后用规则或另一个 LLM 验证结果正确性。
2. **环境反馈**：利用代码执行结果、API 返回状态等客观反馈。
3. **Self-Critique**：让 LLM 自己审查并批评前一步输出。
4. **Reflexion 机制**：失败后反思并利用反思信息重试。
5. **多数投票**：采样多次取一致性最高的答案。
6. **设置最大重试次数**：避免无限纠错循环。

### Q8：什么是 Self-Refine？和 Reflexion 有何区别？（⭐⭐⭐）

**参考答案**：

**Self-Refine**：
- 模型自己生成 → 自己批评 → 自己改进，迭代直到输出满意。
- 不需要外部环境反馈。
- 适用于文本生成优化（如代码优化、文章润色）。

**Reflexion**：
- 需要在真实环境中**执行并获取反馈**。
- 反思结果存入记忆，指导后续尝试。
- 适用于需要与环境交互的 Agent 任务。

**核心区别**：Self-Refine 是"闭卷自评"，Reflexion 是"开卷实战后复盘"。
